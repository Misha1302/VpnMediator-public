# VpnMediator / Razaltush VPN
## Список требуемых UX, product и marketing изменений

**Статус документа:** implementation-ready specification  
**Режим:** Product/UX review + implementation planning  
**Исходный артефакт:** `VpnMediator-ea428d9.tar.gz`  
**Цель:** сделать Telegram-бот понятным неопытному, невнимательному и тревожному пользователю, не ослабляя безопасность, idempotency, stale-state checks и доменную модель подписок/оплат.

---

> **Superseded connection-flow note (2026-06-13):** the current implementation uses one shared subscription URL. The primary action is `Открыть в Happ`; platform selection is optional and only chooses an installation source. Historical per-device-link recommendations below are retained as review history and are not the current runtime specification.

## 1. Главный вывод

Техническая основа сервиса уже сильнее его пользовательского интерфейса. Основные проблемы находятся не в отсутствии функций, а в:

- конкурирующих действиях на одном экране;
- слишком длинном пути от оплаты до первого подключения;
- неточных формулировках статусов;
- скрытых коммерческих функциях;
- несогласованном языке кнопок;
- отсутствии чёткой визуальной иерархии.

Основной принцип изменений:

> На каждом экране пользователь должен видеть одно очевидное главное действие, понимать ожидаемый результат до нажатия и иметь понятный путь восстановления после ошибки.

---

## 2. Архитектурные ограничения

Изменения UX не должны нарушать следующие инварианты:

1. Telegram handler не становится владельцем состояния подписки или заказа.
2. Mediator/domain services остаются source of truth для VPN и subscription state.
3. Нельзя убирать stale quote validation ради упрощения сценария.
4. Нельзя ослаблять idempotency повторных callback и оплат.
5. Нельзя создавать invoice для заказа с итоговой ценой `0`.
6. Нельзя выдавать доступ до допустимого confirmed/completed состояния заказа.
7. Нельзя скрывать техническую ошибку ложным сообщением об успешном VPN-подключении.
8. Нельзя позволять создать второй конкурирующий заказ, пока предыдущий требует продолжения, отмены или повторной активации.
9. Trial eligibility должна по-прежнему проверяться server-side.
10. Реальные credential/token значения не должны попадать в логи, аналитику, тестовые snapshots и документацию.

# P0 — обязательные изменения до публичного запуска

## P0-01. Изолировать незавершённый заказ от новой покупки

### Проблема

`main_menu_keyboard()` сначала строит обычное меню, а затем вставляет сверху действия незавершённого заказа. В результате пользователь может одновременно видеть:

- «Продолжить незавершённый заказ»;
- «Отменить заказ»;
- «Купить доступ»;
- trial CTA;
- другие несвязанные действия.

При `activation_failed` рядом могут находиться «Повторить активацию» и «Купить доступ».

Это создаёт риск повторной покупки и ощущение, что оплаченный заказ потерян.

### Требуемое изменение

При наличии `pending_order_public_id` главное меню должно перейти в отдельный режим заказа и **не добавлять обычные purchase/trial CTA**.

Целевые состояния:

#### PAYMENT_PENDING

Текст:

> **Оплата ещё не завершена**  
> Заказ сохранён. Новую покупку создавать не нужно.

Кнопки:

1. `Продолжить оплату`
2. `Отменить заказ`
3. `Помощь с оплатой`

#### PAYMENT_RECEIVED / PAID, но активация ещё не завершена

Текст:

> **Оплата получена**  
> Мы заканчиваем включение доступа. Повторно оплачивать не нужно.

Кнопки:

1. `Проверить активацию` или `Повторить активацию` — в зависимости от domain state
2. `Помощь`

#### ACTIVATION_FAILED

Текст:

> **Не удалось включить доступ автоматически**  
> Оплата сохранена. Повторно платить не нужно.

Кнопки:

1. `Повторить активацию`
2. `Проблема не решилась`
3. `Главное меню` — только если возврат не показывает новую покупку

### Owning layer

- UI state composition: `VpnAccessBot/vpn_access_bot/home.py`
- Keyboard composition: `VpnAccessBot/vpn_access_bot/keyboards.py`
- Main menu rendering: `VpnAccessBot/vpn_access_bot/handlers/common.py`
- User-facing text: `VpnAccessBot/vpn_access_bot/user_texts.py` или отдельный централизованный text builder

### Рекомендуемая реализация

- Добавить отдельный builder `pending_order_keyboard(state)`.
- В `main_menu_keyboard()` обрабатывать pending-order state первым и завершать построение клавиатуры без добавления стандартных CTA.
- Не дублировать доменную интерпретацию статуса в нескольких handlers.

### Обязательные тесты

- `PAYMENT_PENDING` не содержит `buy:menu` и `trial:*`.
- `ACTIVATION_FAILED` не содержит `buy:menu`.
- Повторное нажатие `continue/retry` не создаёт новый order.
- После успешной активации обычное меню возвращается.
- После отмены только отменяемого заказа снова доступна покупка.

### Acceptance criteria

- На экране незавершённого заказа отсутствует кнопка новой покупки.
- Пользователь явно видит, что повторно платить не нужно.
- Повторный callback идемпотентен.
- Существующие stale-state и ownership checks сохранены.

## P0-02. Не утверждать, что VPN подключён, если подтверждена только загрузка подписки

### Проблема

Фоновое уведомление сообщает:

> VPN подключён. Устройство подтверждено. Теперь можно пользоваться VPN.

Фактически система подтверждает получение/загрузку подписки клиентом, но не обязательно поднятие VPN-туннеля.

### Требуемое изменение

Заменить пользовательское сообщение на честное описание наблюдаемого состояния.

Целевой текст:

> **Подписка добавлена в Happ**  
> Список подключений загружен. Теперь откройте Happ и включите VPN большой кнопкой на главном экране приложения.

Кнопка:

- `Открыть мой VPN` или `Готово`

### Owning layer

- Worker notification: `VpnAccessBot/vpn_access_bot/product_completion.py`
- Общие тексты состояний: `VpnAccessBot/vpn_access_bot/texts.py` / `user_texts.py`

### Обязательные тесты

- В уведомлении нет фразы `VPN подключён`.
- Текст говорит о добавленной подписке, а не о подтверждённом туннеле.
- Notification delivery остаётся idempotent.

### Acceptance criteria

Ни один экран не заявляет о поднятом VPN-туннеле без фактического сигнала, подтверждающего туннель.

## P0-03. Синхронизировать «Как это работает» с текущей credential delivery model

### Проблема

Инструкция описывает устаревший flow через временную страницу и временную ссылку, тогда как текущий сценарий выдаёт отдельную ссылку устройства и позволяет:

- скопировать её;
- открыть deep link Happ, если доступен;
- повторно показать;
- перевыпустить;
- отозвать.

### Требуемое изменение

Сделать инструкцию единой и соответствующей текущему коду.

Целевой текст:

> **Как подключить Razaltush VPN**
>
> 1. Выберите устройство.
> 2. Установите Happ, если его ещё нет.
> 3. Нажмите «Добавить в Happ» или скопируйте ссылку и вставьте её в раздел подписок Happ.
> 4. Откройте Happ и включите VPN.
>
> Для каждого устройства бот создаёт отдельную ссылку. Не пересылайте её другим людям.

### Owning layer

- `VpnAccessBot/vpn_access_bot/texts.py`
- `VpnAccessBot/vpn_access_bot/handlers/common.py`
- `docs/user-guide.md`
- `VpnAccessBot/README.md`

### Обязательные тесты

- Text tests не содержат описания удалённой временной страницы.
- Документация и bot text используют одинаковые названия кнопок.
- Все callback, упомянутые в инструкции, реально существуют.

### Acceptance criteria

Инструкция, фактическая клавиатура и документация описывают один и тот же flow.

## P0-04. Убрать обязательную ручную проверку получения из happy path

### Проблема

После выдачи ссылки пользователь видит кнопку «Проверить получение» и текст, требующий нажать её. При этом background worker уже умеет обнаруживать загрузку подписки.

Новичок не понимает, что именно проверяется, и может считать, что VPN не работает без этого действия.

### Требуемое изменение

- Не требовать ручную проверку в основном тексте.
- Оставить её как вторичное диагностическое действие, например `Не появилось в Happ?`.
- После выдачи ссылки показать единственный главный следующий шаг: открыть Happ и включить VPN.
- Автоматическое уведомление использовать как основной механизм подтверждения получения.

Целевой экран после выдачи:

> **Добавьте подписку в Happ**  
> Нажмите «Добавить в Happ». Если кнопка не сработала, скопируйте ссылку вручную.
>
> После добавления откройте Happ и включите VPN.

Кнопки:

1. `Добавить в Happ` — если deep link поддерживается
2. `Скопировать ссылку`
3. `Не появилось в Happ?`
4. `Главное меню`

### Owning layer

- `VpnAccessBot/vpn_access_bot/keyboards.py`
- `VpnAccessBot/vpn_access_bot/handlers/onboarding.py`
- `VpnAccessBot/vpn_access_bot/product_completion.py`

### Обязательные тесты

- Happy path не требует callback `handoff:check`.
- Диагностическая ручная проверка остаётся доступной.
- Background completion notification не дублируется.
- Повторное открытие экрана не создаёт новую credential без явного перевыпуска.

### Acceptance criteria

Пользователь может завершить основной flow, не понимая термина «проверка получения» и не нажимая ручную техническую проверку.

# P1 — основной UX-редизайн

## P1-01. Сократить onboarding до трёх основных действий

### Текущий путь

1. Подключить устройство.
2. Выбрать платформу.
3. Открыть установку Happ.
4. Вернуться в Telegram.
5. Нажать «Happ уже установлен».
6. Нажать «Добавить Razaltush VPN в Happ».
7. Получить credential.
8. Добавить/copy.
9. Нажать ручную проверку.

### Целевой путь

1. `Подключить устройство`.
2. Выбрать платформу.
3. На одном экране установить Happ при необходимости и получить ссылку.

### Требуемое изменение

После выбора платформы показать комбинированный экран:

> **Подключение на Android — шаг 2 из 2**
>
> 1. Установите Happ, если приложение ещё не установлено.
> 2. Добавьте подписку в Happ.

Кнопки:

1. `Установить Happ`
2. `Добавить в Happ` или `Получить ссылку`
3. `Другой способ установки` — при наличии
4. `Не получилось`
5. `Назад`

Допускается оставить внутренний переход между install и credential generation, но пользователь не должен проходить отдельный экран «Happ уже установлен», если credential можно безопасно подготовить сразу.

### Важное ограничение

Нельзя заранее расходовать slot или создавать credential, если пользователь только открыл список платформ и ещё не подтвердил подключение. Credential generation должна оставаться явным действием.

### Owning layer

- `handlers/onboarding.py`
- `keyboards.py`
- `credential_delivery.py`
- `client_catalog.py`

### Acceptance criteria

- От активной подписки до кнопки copy/deep-link — не более трёх основных нажатий.
- Нет обязательного промежуточного экрана `Happ уже установлен`.
- Повторный вход продолжает существующую onboarding session.
- Новая credential не создаётся при простом открытии меню.

## P1-05. Объяснить тарифы через сценарии использования

### Проблема

Количество устройств выбирается голыми числами, например `1`, `2`, `3`, `5`, `8`, `12`.

### Требуемое изменение

Основные варианты:

- `1 устройство — только телефон`
- `3 устройства — телефон + компьютер · Популярно`
- `5 устройств — для семьи`
- `Другое количество`

В дополнительном меню можно оставить 2, 8 и 12 устройств.

### Периоды

После выбора количества устройств кнопки периода должны показывать конечную цену:

- `1 месяц — 60 ⭐`
- `3 месяца — 171 ⭐ · экономия 5%`
- `6 месяцев — 324 ⭐ · экономия 10%`
- `12 месяцев — 612 ⭐ · выгоднее всего`

Значения должны вычисляться из реального quote/catalog, а не быть захардкожены в UI.

### Owning layer

- `product_catalog.py`
- `commerce.py`
- `handlers/buy.py`
- `keyboards.py`

### Обязательные тесты

- Цена на кнопке совпадает с итоговым quote.
- Discount label соответствует фактической экономии.
- Изменение catalog не требует ручной правки текста цены.
- Upgrade/renew используют те же price builders.

## P1-07. Унифицировать словарь интерфейса

### Требуемый словарь

Использовать единообразно:

- `← Назад`
- `🏠 Главное меню`
- `Помощь`
- `Мой доступ`
- `Мои устройства`
- `Продлить доступ`
- `Подключить устройство`
- `Увеличить число устройств`
- `Попробовать 2 дня бесплатно`

Не смешивать:

- `Назад`, `⬅️ Назад`, `⬅️ Меню`;
- `В главное меню`, `⬅️ В главное меню`;
- `Помощь`, `Поддержка` на одинаковом уровне;
- `Мой VPN`, `Подписка`, `Информация о VPN` для одного и того же раздела.

### Статусы

- `🟢 Доступ активен`
- `🟡 Нужно закончить подключение`
- `🔴 Доступ закончился`
- `⚠️ Не удалось включить доступ`

### Реализация

- Вынести labels в централизованный модуль/набор constants, если это не создаёт декоративную абстракцию.
- Не размазывать одинаковые строки по handlers.

### Тесты

- Text/keyboards tests проверяют отсутствие устаревших вариантов.
- Navigation callbacks остаются прежними там, где можно избежать breaking changes.

## P1-08. Упростить предупреждение о секретной ссылке

### Проблема

Безопасность объясняется многократно и тревожными словами: «секретная ссылка», «токен», «удалите сообщение» и т. п.

### Требуемое изменение

Основная спокойная формулировка:

# P2 — конверсия, retention и визуальная полировка

## P2-01. Улучшить первый экран нового пользователя

### Целевой текст

> **Razaltush VPN**
>
> Простой VPN для телефона и компьютера.
>
> Попробуйте 2 дня бесплатно — без карты. Подключение занимает несколько минут, а при проблеме можно написать прямо в помощь.

Кнопки:

1. `Попробовать 2 дня бесплатно`
2. `Выбрать срок`
3. `Как подключается VPN`

### Требования

- Не использовать `Подключить бесплатно`, поскольку это может восприниматься как бессрочный бесплатный доступ.
- Не обещать автоматическое подключение, если требуется установка Happ.
- Не обещать абсолютную доступность во всех странах/сетях без подтверждения.

## P2-02. Показывать referral после полученной ценности

### Требуемое изменение

Не продвигать referral до первого успешного добавления подписки.

После completion notification добавить вторичную кнопку:

- `Подарить другу VPN`

Текст referral:

> Друг оплатит 30 дней — вам добавится 6 дней.  
> Оплатит 6 месяцев — вам добавится примерно 36 дней.

Формула и примеры должны соответствовать реальному reward policy.

### Тесты

- Reward preview совпадает с domain calculation.
- Нельзя получить reward за самого себя, повторную связь или неоплаченный заказ.

## P2-03. Ввести продуктовую аналитику воронки

### Минимальные события

- `bot_started`
- `trial_cta_opened`
- `trial_activated`
- `purchase_started`
- `devices_selected`
- `period_selected`
- `quote_shown`
- `payment_started`
- `payment_completed`
- `activation_completed`
- `connect_device_started`
- `platform_selected`
- `credential_created`
- `credential_copied` — только факт действия, без URL
- `happ_deep_link_opened`
- `subscription_observed_by_client`
- `support_opened`
- `device_limit_reached`
- `renewal_started`
- `upgrade_started`

### Обязательные свойства

- user/subscription/order identifiers только в безопасной внутренней форме;
- credential URL и токены никогда не входят в payload;
- события не становятся второй source of truth;
- повторные callback не раздувают метрики без idempotency key.

### Основные метрики

- start → trial/purchase conversion;
- payment → activation conversion;
- activation → credential creation;
- credential creation → subscription observed;
- median time to first connected device;
- support-open rate по каждому шагу;
- renewal conversion;
- upgrade conversion;
- повторные нажатия и abandon points.

## P2-04. Провести usability test на реальных новичках

### Выборка

Минимум пять человек, которые:

- раньше не пользовались Happ;
- не знают архитектуру VPN;
- выполняют сценарий на собственном телефоне;
- не получают подсказок от разработчика.

### Сценарии

1. Активировать trial.
2. Подключить телефон.
3. Понять, что делать после добавления подписки.
4. Найти срок действия.
5. Продлить доступ.
6. Освободить занятое место устройства.
7. Найти помощь после ошибки.

### Критерии

- не менее 90% завершают первый flow без помощи оператора;
- пользователь не создаёт повторный заказ;
- пользователь различает «подписка добавлена» и «VPN включён»;
- до credential delivery не более трёх основных нажатий;
- ни один участник не спрашивает, что такое `handoff`, `credential`, `upstream`, `catalog` или `token`.

# 3. Целевые пользовательские сценарии

## 3.1. Новый пользователь

1. Видит ценность и trial без карты.
2. Нажимает `Попробовать 2 дня бесплатно`.
3. Подтверждает понятные условия.
4. После активации видит одну главную кнопку `Подключить устройство`.
5. Выбирает платформу.
6. Устанавливает Happ при необходимости и получает ссылку на одном экране.
7. Добавляет подписку.
8. Получает честное уведомление `Подписка добавлена в Happ`.
9. Включает VPN внутри Happ.

## 3.2. Активный пользователь

Главный экран сразу показывает:

- дату окончания;
- `X из Y` устройств;
- подключение нового устройства;
- устройства;
- продление;
- меню `Ещё`.

## 3.3. Продление

1. Пользователь нажимает `Продлить доступ` из меню или уведомления.
2. Выбирает период.
3. Видит конечную цену и новую дату окончания.
4. Подтверждает оплату.
5. Повторный callback не создаёт второе продление.

## 3.4. Увеличение числа устройств

1. Пользователь нажимает `Увеличить число устройств` из `Ещё` или recovery screen лимита.
2. Видит текущий лимит и новые варианты.
3. Видит конечную доплату и результат `станет X устройств`.
4. После оплаты лимит обновляется один раз.

## 3.5. Ошибка оплаты или активации

1. Бот явно сообщает, сохранена ли оплата.
2. Не предлагает новую покупку.
3. Даёт одно главное recovery action.
4. Содержит прямую кнопку помощи.
5. После устранения ошибки возвращает пользователя в нормальное состояние.

## 3.6. Повторное нажатие

- повторный callback не создаёт duplicate order;
- повторная выдача существующей credential не меняет её;
- перевыпуск выполняется только после явного подтверждения;
- repeated notification не отправляется повторно;
- пользователь всегда получает понятный текущий state, а не generic error.

# 6. Общие acceptance criteria

UX-редизайн считается принятым, когда выполнены все условия:

1. На каждом обычном экране есть одно визуально главное действие.
2. Незавершённый заказ не соседствует с новой покупкой или trial.
3. Ни один текст не утверждает поднятие VPN-туннеля без фактической проверки.
4. Инструкция соответствует текущей credential delivery model.
5. Ручная проверка получения не обязательна для happy path.
6. От активированной подписки до credential delivery — не более трёх основных нажатий.
7. Upgrade и referral доступны через кнопки без slash-команд.
8. Периоды показывают конечную цену из реального quote.
9. Уведомление о продлении содержит прямой renew CTA.
10. Навигационные подписи и термины единообразны.
11. Credential security не ослаблена.
12. Duplicate callbacks не создают duplicate orders, credentials или lifecycle mutations.
13. Все focused и full relevant tests проходят.
14. Документация описывает фактически реализованный flow.
15. Usability test подтверждает, что минимум 90% новичков завершают первый сценарий без помощи оператора.

# 8. Что не входит в данный scope

Этот документ не требует:

- замены VPN upstream;
- переработки ASP.NET Mediator architecture;
- отмены Happ как клиентского приложения;
- ослабления token/credential security;
- отключения stale quote checks;
- изменения payment provider без отдельного бизнес-требования;
- редизайна логотипа как замены UX-исправлениям;
- добавления серверного подтверждения реального туннеля, если upstream пока не предоставляет такой сигнал.

Если в будущем появится надёжный signal реального VPN tunnel health, можно добавить отдельное состояние `VPN действительно подключён`; до этого интерфейс должен ограничиваться подтверждёнными фактами.
